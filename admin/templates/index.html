Restrict users to specific plants (USERS_JSON)

Operators can be restricted to see only their plant's data. Set the USERS_JSON environment variable in .env or docker-compose.yml:

USERS_JSON={"admin":{"password":"strongPass","role":"admin"},"op-sp":{"password":"1234","role":"operator","plants":["SP-Factory"]},"op-rj":{"password":"5678","role":"operator","plants":["RJ-Plant"]},"monitor":{"password":"monitor","role":"monitor"}}
UserRolePlantsResult
adminadmin(all)Sees and configures everything
op-spoperatorSP-FactoryOnly sees devices in SP-Factory, can write to PLC
op-rjoperatorRJ-PlantOnly sees devices in RJ-Plant, can write to PLC
monitormonitor(all)Sees all plants, read-only + alarms + logbook

The plants field is optional. Users without it see all plants. The plant name must match the Plant / Site field configured in each device.

  1. Add devices: Go to Devices tab and click + Add Device, or click Load Demo Devices for examples
  2. Restart the server: Click Restart in the top right after adding/removing devices
  3. Monitor live data: Go to Live Data tab to see real-time tag values
  4. View dashboards: Open Grafana for time-series graphs, trends, and AI predictions
  5. Monitoring room TV: Open Grafana with ?kiosk for full screen, or use the Monitoring Room playlist to auto-rotate dashboards every 30s
  6. Connect OPC-UA clients: Point your SCADA/HMI to opc.tcp://<this-server>:12687/plc4x
  7. Integrate via MQTT: Subscribe to plc4x/# on port 1883 for real-time tag values
  8. Use the API: Go to the API tab for Swagger documentation with "Try it out"
  9. Custom data: Send data from any source (manual inputs, vision AI, old equipment) via POST /api/data/write — see examples below
  10. Create HMI screens: Go to HMI tab, click Load Demo for examples, or create a plant/area/equipment hierarchy and drag elements onto the canvas. Use Fullscreen for TV monitors or add ?hmi=equip-id to the URL.
Data Ingestion Examples (POST /api/data/write)

Send any data to InfluxDB via the REST API. It appears in Grafana alongside PLC data. Supports single or batch writes.

Quality Inspection (manual): 
curl -X POST http://<host>:8080/api/data/write \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "measurement": "quality_check",
    "tags": {"line": "Line1", "product": "BoxA", "inspector": "operator1"},
    "fields": {"result": 1, "score": 0.95}
  }'
Computer Vision AI: 
curl -X POST http://<host>:8080/api/data/write \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "measurement": "vision_ai",
    "tags": {"camera": "cam01", "model": "defect_v2"},
    "fields": {"defect_detected": 0, "confidence": 0.98, "inference_ms": 45}
  }'
Old Equipment (manual reading): 
curl -X POST http://<host>:8080/api/data/write \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "measurement": "manual_reading",
    "tags": {"equipment": "OldBoiler1", "operator": "silvio"},
    "fields": {"temperature": 85.5, "pressure": 2.3}
  }'
Batch (multiple points): 
curl -X POST http://<host>:8080/api/data/write \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '[
    {"measurement": "production", "tags": {"line": "L1"}, "fields": {"count": 150, "reject": 3}},
    {"measurement": "production", "tags": {"line": "L2"}, "fields": {"count": 200, "reject": 1}}
  ]'

All data is stored in InfluxDB and visible in Grafana. Also published to MQTT at plc4x/_custom/{measurement}. Full docs in the API tab (Swagger).

▶ How It Works — Algorithm Reference

Anomaly Detection — PyOD Ensemble

Three algorithms run simultaneously on each tag: ECOD (statistical distribution), LOF (local density), and Isolation Forest (tree partitioning). A point is flagged as anomalous only when at least 2 of 3 algorithms agree — reducing false positives. Confidence is reported as a percentage.

Tune with: Contamination (expected % anomalies) and Min Agreement (1=sensitive, 3=strict).

SHAP Explainability

SHAP (SHapley Additive exPlanations) answers "why was this flagged?". It trains a gradient boosting model on tag history and computes the contribution of each feature (tag) to the anomaly prediction. The top contributors are shown as colored bars — positive means the tag pushed toward anomaly, negative means it pulled away.

Tune with: Top Contributors (how many tags to show).

Cross-Tag Correlation — Statsmodels

Pearson correlation is computed between all numeric tag pairs. A broken correlation alert fires when a pair that was strongly correlated (e.g., 0.85) drops below the break threshold — indicating a relationship changed, which often precedes failures. The correlation matrix shows all pair relationships at a glance.

Tune with: Break Threshold (lower = more sensitive alerts).

Change Point Detection — ruptures

Uses the PELT algorithm (Pruned Exact Linear Time) to detect abrupt changes in the statistical properties of a signal — mean, variance, or both. Each detected change point includes the mean before and after, and a severity score based on the magnitude of change. Useful for detecting process regime shifts.

Tune with: Penalty (higher = fewer detected changes, lower = more sensitive).

Pattern Matching — stumpy (Matrix Profile)

stumpy computes the Matrix Profile — a compact data structure that reveals the most similar subsequences (motifs) and the most unusual subsequences (discords) in a time series. Motifs show recurring patterns; discords flag unusual behavior. Similarity is reported as a percentage (100% = exact match).

Tune with: Window Size (length of pattern to search) and Top K (how many patterns to return).

Time-Series Forecast (Prophet)

Prophet (by Meta) predicts future tag values based on historical trends and seasonality. It generates a forecast for the next 2 hours with upper and lower confidence bounds. When the actual value falls outside these bounds, it may indicate an unexpected process change. The forecast is updated every cycle and visualized in Grafana.

Tune with: Forecast Hours (how far ahead to predict). Needs 100+ data points (about 8 minutes at 5s polling).

Trend Analysis — NumPy (Linear Regression)

Fits a linear regression to the last 24 hours of data and calculates the rate of change per hour. A rate of +5.2 means the value is increasing by 5.2 units every hour. This helps identify slowly drifting processes that are hard to spot in real-time — for example, a bearing temperature rising 0.3 degrees per hour over a shift.

Output: rate_per_hour in InfluxDB. Positive = rising, negative = falling, near zero = stable.

Classic Anomaly Detection — scikit-learn (Isolation Forest)

The original single-algorithm anomaly detector using scikit-learn Isolation Forest. It scores each tag value based on how easy it is to "isolate" from the rest of the data — outliers are isolated quickly and get high anomaly scores. This runs alongside the PyOD ensemble as an additional perspective. Results are written to InfluxDB for historical anomaly tracking.

Note: The PyOD ensemble (above) also includes Isolation Forest as one of its 3 algorithms, providing a combined view.

Alert Severity Levels

Critical (red) — Anomaly detected, action required
Warning (orange) — Correlation broke, investigate
Info (blue) — Change point or pattern found
Tips: Start with defaults. Increase contamination if too many false positives appear. Raise penalty for change points if results are noisy. All results are also available in the Analytics tab via the "ML Insights" Grafana dashboard.
